### AI Product Manager Portfolio Project | Hong Kong Brokerage & Fintech

## 项目背景

这是一个 AI 产品经理作品集项目，模拟香港持牌券商 App 内嵌 AI 交易助理的产品概念验证（PoC）。

**我的背景**：在香港传统券商担任交易系统产品经理，负责柜台系统与客户端产品设计。这个项目是我将券商业务经验与 AI 产品设计能力结合的实践成果。

**核心问题**：香港零售投资者在交易决策中面临信息不对称——财务数据散落各处，行情难以解读，新手投资者尤其缺乏基础金融教育。传统券商 App 提供的是数据展示，不是理解辅助。

**产品假设**：在持牌券商 App 内嵌入 AI 助理，能够在合规边界内降低投资者的信息获取门槛，同时不越界提供受监管的投资建议服务。

---

## 产品设计决策

这个 demo 背后的每一个设计选择都有产品逻辑，而不是技术便利。

### 决策一：明确定义 AI 的边界，而不是扩大能力

**问题**：在金融场景部署 AI，最危险的不是 AI 不够聪明，而是 AI 越界做了不该做的事。香港《证券及期货条例》明确规定，提供投资建议属于受规管活动，须持有第 4 类牌照。

**设计选择**：
- 构建关键词级别的边界检测层（`detectBoundaryViolation()`），覆盖中英文投资建议请求
- AI 拒绝回答时，主动提供替代价值（"我可以帮你分析基本面数据"）
- 在拒绝回答旁展示"寻找持牌顾问"按钮，引导合规路径

**取舍**：这个设计会降低部分用户的即时满足感，但避免了监管风险。产品的长期可持续性优先于短期参与度。

---

### 决策二：双语支持（中文优先，英文并行）

**问题**：香港投资者群体语言习惯多元——本地用户倾向粤语/普通话，机构和专业投资者倾向英语。单语产品会天然排除一部分用户。

**设计选择**：
- 完整的 i18n 架构，中英文各自维护独立的文案库（系统提示词、UI 文案、demo 回复全部双语）
- AI 自动检测用户输入语言并以同语回复（`hasCJK()` 检测 + 语言传递至 system prompt）
- 语言切换不中断对话上下文

**产品洞察**：语言不只是 UI 问题，系统提示词的语言会直接影响 AI 的回复质量和文化语境准确性。中文 system prompt 生成的中文回复比英文 system prompt 更自然。

---

### 决策三：离线 Demo 模式（无需 API Key 即可体验）

**问题**：在面试、产品评审、或向非技术利益相关者演示时，依赖实时 API 调用存在风险——网络延迟、API 配额、Key 泄露都是潜在问题。

**设计选择**：
- 内置完整的 demo 回复库，覆盖最高频的 5 类用户问题
- 跳过 API Key 时自动切换至 demo 模式，体验流程不中断
- Demo 回复与真实 AI 回复在界面上无差异，保证演示效果一致

**类比**：这是产品经理常说的"Wizard of Oz"原型——在验证用户行为和界面设计时，不需要真实 AI 在后台运行。

---

### 决策四：对话上下文管理（Context Window 控制）

**问题**：维持多轮对话需要将完整历史传给 API，但 token 数量无限增长会导致成本上升和响应变慢。

**设计选择**：
- 保留 system prompt + 最近 19 条对话记录（滑动窗口）
- System prompt 包含完整组合数据，确保 AI 每次回复都有足够的用户上下文
- 超出窗口的历史自动裁剪，用户无感知

**产品权衡**：19 条对话约覆盖一次完整的咨询会话，超出部分对大多数用户场景无影响。对于需要更长上下文的场景（如复杂投资组合分析），这是已知局限，后续可通过对话摘要机制解决。

---

### 决策五：合规信息披露的视觉层级

**问题**：监管要求的风险声明往往被设计成视觉噪音，用户习惯性忽略。但这些声明在法律上是必须的。

**设计选择**：
- 每条 AI 回复附带轻量级 compliance badge（"ℹ️ 信息用途"），而非底部静态声明
- Badge 与消息绑定，强调每一条 AI 输出都是信息，不是建议
- 页面底部保留完整法律声明（监管合规要求）
- 两层设计：用户体验层（badge，轻量）+ 合规层（底部声明，完整）

## 技术实现

| 组件 | 实现方式 | 选择理由 |
|------|---------|---------|
| 前端 | 单文件 HTML + 原生 JS | 零依赖，任何环境可运行，易于演示 |
| AI 后端 | DeepSeek API（OpenAI 兼容） | 成本低，中文能力强，适合香港金融场景 |
| 对话管理 | 客户端滑动窗口（19轮） | 控制 token 成本，保持上下文连贯 |
| 离线模式 | 内置 demo 回复库 | 演示稳定性，降低 API 依赖 |
| 合规检测 | 正则表达式关键词匹配 | 轻量、可解释、易于扩展规则 |

**System Prompt 设计原则**：
```
角色定义 → 数据上下文（组合数据）→ 能力边界（能做什么）→ 硬性限制（不能做什么）→ 语气要求
```
这个顺序很重要——先告诉 AI 它是谁，再给数据，再定规则。顺序颠倒会降低指令遵循率。

## 已知局限与下一步迭代

这个 PoC 有意省略了一些功能，以下是产品路线图思考：

**短期（如果进入 Beta）**：
- 接入实时行情 API（替换模拟数据）
- 对话记录持久化（用户端加密存储）
- 覆盖更多合规边界场景（衍生品、杠杆产品的处理逻辑）

**中期（规模化前必须解决）**：
- 合规审计日志：每条 AI 回复与用户问题的完整记录，供监管查验
- 对话摘要机制：解决长会话的上下文窗口限制
- 人工接管触发器：当 AI 置信度低时，主动转接人工客服

**需要验证的核心假设**：
- 用户是否真的接受 AI 拒绝提供投资建议，还是会直接流失？
- "找持牌顾问"的转化率是否足够高，能支撑顾问团队的 CAC？
- 双语切换是否会影响用户的信任感？
